Introduction

JMVC implements the MVC pseudo-pattern using vanilla javascript. You can write down your Controllers with your actions. Within the action you can load one or more of your Models, get data from server using asynchronous xhr calls, load one or more Views and at the end render the View tree, populated with data retrieved.

Any page under the jmvc.org domain relies on JMVC, even the 404 page. JMVC provides a bunch of popular and less popular features within a clean namespace, allowing to make ajax calls, inspect, traverse and edit the DOM tree, manage events, and exposing useful functions on javascript objects.

Moreover, it gives You a heap of OOP optimized functions and constructors, to speed up any suitable work an turn it into an extremely funny experience.
The repository is filled out of extensions which provides many useful ready to use objects and widgets.

Requirements

At the moment, JMVC relies completely on Apache webserver with rewrite module enabled. In fact the single page entry point is granted by an .htaccess file. With IIS a .webconfig file can be used to get the same work done.
So at least are needed

  • web server with a basic url-rewrite mechanism OR nodejs
  • text editor
  • an awake browser

As alternative absolutely only for development fire up the inner node minimal dummy server

  • run node server.js from the srv/ folder
  • add foo.dev 127.0.0.1 to your hosts file
  • edit the srv/config.json file to fill out the true path for the webroot
  • visit http://www.foo.dev:8080

If you feel more confortable with port 80, change port setting in the srv/config.json file, but remember that if port is under 1024, only root will start the server successfully.

Get started

Get the source, create a simple virtual host (or fireup node srv), and start playing around. Note that the repository code is exactly this site. To clean all just remove everything within the

  • app/models/ folder
  • app/i18n/ folder
  • app/views/ folder BUT the app/views/core/ folder
  • app/controllers/ folder BUT the app/controllers/404.js controller
  • media/ folder BUT the media/css/reset.css and media/css/core/jmvc.min.css files

Now provide at least an index controller with a action_index function within and visit your virtual host root.
I strongly recommend to keep all files, look at the code, starting from controllers, and clean up when You feel ready to dive in.

How JMVC works, distilled

JMVC relies on a single entry page, the index.html, thanks to a minimal .htaccess file. In that page the main script looks at the requested url and the result of this operation is that JMVC knows which controller must be loaded, wich action must be called and what parameters have been passed (dispatching).
Once the Controller is loaded, with a synchronous xhr call, the action function is executed, and within that function at least a view is supposed to be rendered in the document body (or elsewhere).

Url structure

Supposing we`re using virtual host http://www.jmvc.dev , a location will be something like the following:

http://www.jmvc.dev/ Cnt/ Act/ n1 / v1 / n2 / v2 / n3 ?v4=Fede&v5=ghe

  • : http://www.jmvc.dev
  • : Cnt
  • : Act
  • : n1/v1/n2/v2/n3
  • : ?v4=Fede&v5=ghe

All parameters (including those ones coming from querystring) will be automatically loaded into the current controller register, meaning that any view rendered from it can use the placeholders to show their value.

HINT: parameters are meant to be even, in the controller You will be able to access and use 2 variables: n1="v1" and n2="v2"; the third one will be ignored because unvalued.
