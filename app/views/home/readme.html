Description

JMVC implements the MVC pseudo-pattern using only pure vanilla javascript. You can write down your Controllers with your actions. Within the action you can load one or more of your Models, get data from server using asynchronous xhr calls, load one or more Views and at the end render the View tree, populated with data retrieved.

Any page under the jmvc.org domain relies on JMVC, even the 404 page. But that`s not all at all! JMVC provides a buch of popular and less popular features within a clean namespace of libraries, allowing to make ajax calls, inspect, traverse and edit the DOM tree, manage events, and exposing useful functions on javascript objects.

Moreover, it gives You a heap of OOP optimized functions, to speed You up, and making any suitable work an extremely funny experience.
But there`s even more: the repository is filled of extensions which provides many useful ready to use objects and widgets (some of wich are still under development).

Requirements

At the moment, JMVC relies completely on Apache webserver with rewrite module enabled. In fact the single page entry point is granted by an .htaccess file.

Get started

Get the source here, create a simple virtual host, and start playing around. Note that the code is exactly this site. To clean all just remove everything within the

  • app/models/ folder
  • app/i18n/ folder
  • app/views/ folder BUT the app/views/core/ folder
  • app/controllers/ folder BUT the app/controllers/404.js controller
  • app/media/ folder BUT the app/media/core/ folder

Now provide at least an index controller with a action_index function within and visit your virtual host root.
I strongly recommend to keep all files, look at the code, starting from controllers, and clean up at when You feel ready to dive in.

How it works

JMVC relies on a single entry page, the index.html, thanks to a minimal .htaccess file. In that page the main script looks at the requested url and the result of this operation is that JMVC knows which controller must be loaded, wich action must be called and what parameters have been passed. This phase is called dispatching.
Once the controller is loaded, with a synchronous xhr call, the action function is executed, and within that function at least a view is supposed to be rendered in the document body (or eslewhere).

Url structure

Supposing we`re using virtual host http://www.jmvc.dev , a location will be something like the following:

http://www.jmvc.dev/Cnt/Act/n1/v1/n2/v2/n3?v4=Fede&v5=ghe

  • : http://www.jmvc.dev
  • : Cnt
  • : Act
  • : n1/v1/n2/v2/n3
  • : ?v4=Fede&v5=ghe

All parameters (including those ones coming from querystring) will be automatically loaded into the current controller register, meaning that any view rendered from it can use the placeholders to show their value.

HINT: parameters are meant to be even, in the controller You will be able to access and use 2 variables: n1="v1" and n2="v2"; the third one will be ignored because unvalued.
